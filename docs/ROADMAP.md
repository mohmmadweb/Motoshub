# نقشه‌راه پیاده‌سازی موتوشاب (نسخه‌ی یکپارچه‌ی Motoshub + Mizito + Mattermost)

> این سند ادامه‌ی [`ARCHITECTURE.md`](ARCHITECTURE.md) است و فرض می‌گیرد محصول نهایی، ترکیبی از سه پیکره است:
> **۱) هسته‌ی اجتماعی Motoshub** (گروه/انجمن/رویداد/بلاگ/رسانه/اخبار) · **۲) موتور کاری Mizito-گونه** (پروژه/تسک/دانش/گزارش) · **۳) ارتباطات درون‌سازمانی در سطح Mattermost** (کانال، رشته‌ی پاسخ، حضور، یکپارچه‌سازی، انطباق).

---

## ۰. تصمیم معماری حیاتی — قبل از هرچیز باید گرفته شود

ساختن یک موتور چت در سطح Mattermost (رشته‌ی پاسخ پایدار، حضور بلادرنگ، جستجوی کامل پیام‌ها، انطباق/eDiscovery، مقیاس‌پذیری) **از صفر**، خودش یک پروژه‌ی چندساله است. دو مسیر واقع‌بینانه وجود دارد:

| مسیر | توضیح | زمان | ریسک |
|---|---|---|---|
| **الف) self-host کردن خود Mattermost** (پیشنهاد من) | نسخه‌ی متن‌باز Mattermost را به‌عنوان «سرویس ارتباطات» در زیرساخت خودتان بالا می‌آورید، با SSO مشترک (Keycloak) آن را به هویت کاربران موتوشاب وصل می‌کنید، تم/رنگ آن را با CSS override به برند موتوشاب نزدیک می‌کنید، و رابط آن را داخل iframe/پنجره‌ی همان دامنه embed می‌کنید. هر «تننت موتوشاب» = یک «Team» در Mattermost (این مفهوم از قبل در خود Mattermost برای همین کار ساخته شده). | ۳ تا ۵ هفته برای راه‌اندازی + یکپارچه‌سازی | کم — محصول بالغ، تست‌شده، با پشتیبانی فعال |
| ب) بازنویسی کامل چت با زیرساخت موجود (Ratchet/RabbitMQ) | همان چیزی که در پروتوتایپ شبیه‌سازی شده را واقعاً پیاده می‌کنید: کانال، رشته، حضور، reaction. | ۱۲ تا ۱۸ هفته فقط برای رسیدن به سطح فعلی پروتوتایپ، بدون انطباق/eDiscovery واقعی | بالا — قابلیت‌های Mattermost (جستجوی کامل، اسکیل، موبایل، انطباق) سال‌ها زمان برده‌اند |

**توصیه:** مسیر (الف). انرژی تیم را روی چیزی بگذارید که رقبا (میزیتو، اسلک) ندارند: ادغام کامل بستر اجتماعی + کاری + ارتباطی پشت یک هویت و یک تجربه‌ی کاربری، نه روی بازآفرینی یک محصول بالغ که از قبل متن‌باز است.

## ۱. فازبندی کلی (حدود ۶ تا ۷ ماه برای نسخه‌ی واقعاً کامل و قابل‌فروش)

نکته‌ی صادقانه: بازه‌ی ۳ ماهه‌ی اولیه‌ای که در ابتدای این گفتگو مطرح شد، فقط فاز ۱ قرارداد را پوشش می‌داد. حالا که scope شامل سطح Mizito + سطح Mattermost هم شده، **۳ ماه برای نسخه‌ی کامل واقع‌بینانه نیست.** در ادامه فازبندی واقعی آمده؛ اگر فشار زمانی ۳ ماهه واقعاً ثابت است، در پایان سند یک نسخه‌ی «MVP فشرده» هم آورده‌ام.

### فاز ۰ — پایه (۲ تا ۳ هفته)
- راه‌اندازی API Gateway (Kong/nginx) و الگوی تننت‌رزولوشن (دامنه/ساب‌دامین → tenant_id)
- راه‌اندازی Keycloak به‌عنوان IdP مرکزی (SSO/LDAP/OIDC sosial-login) — همه‌ی سرویس‌ها (Oxwall، سرویس‌های Node، Mattermost) به همین IdP اعتماد می‌کنند
- self-host کردن Mattermost + پایگاه‌داده‌ی آن، تنظیم اولیه‌ی برندسازی/تم
- CI/CD، محیط staging، Infrastructure-as-Code (Terraform/Ansible) برای تکرارپذیری استقرار هر تننت
- پیاده‌سازی Design System این پروتوتایپ به‌صورت یک کتابخانه‌ی کامپوننت قابل‌استفاده‌ی مجدد

### فاز ۱ — هسته‌ی اجتماعی + ارتباطات پایه (۸ تا ۱۰ هفته)
- لایه‌ی API REST جدید روی `bol/`های موجود Oxwall برای: کاربران، گروه، پست، انجمن، اعلان، رویداد، بلاگ، رسانه، اخبار، جستجو (Elasticsearch)
- یکپارچه‌سازی Mattermost: نگاشت تننت ↔ Team، SSO مشترک، embed در `/app/chat`، فعال‌سازی Webhook/Slash Command/Bot از طریق API رسمی Mattermost (نه بازسازی)
- چندمستأجری و برندسازی واقعی (دامنه/رنگ/لوگو per-tenant) + پنل راهبری نسخه‌ی واقعی (نه فقط UI نمایشی)
- **خروجی این فاز: نسخه‌ای قابل ارائه به اولین مشتری پایلوت (هلدینگ دانشمند) با فاز ۱ کامل قرارداد + ارتباطات در سطح Mattermost**

### فاز ۲ — دانش و پروژه (۶ تا ۸ هفته)
- مدیریت دانش واقعی: آپلود/دسته‌بندی/نسخه‌گذاری سند، اتصال Elasticsearch برای جستجوی پیشرفته
- موتور پروژه‌ی سطح Mizito: بورد، گانت با وابستگی بین تسک‌ها، بودجه‌بندی، صورت‌جلسات
- گزارش‌گیری پیشرفته‌ی واقعی (aggregation queries، نه داده‌ی mock)
- RBAC کامل (Platform/Org/Group) + پروفایل سازمانی واقعی

### فاز ۳ — ماژول‌های تخصصی فاز ۳ قرارداد + سخت‌سازی (۶ تا ۸ هفته)
- مدیریت قراردادهای فناورانه، صندوق نوآوری/شتاب‌دهی، فرصت‌های پژوهشی — به‌صورت ماژول‌های نصب‌شدنی مجزا (نه core)، چون این‌ها صرفاً برای مشتریانی مثل هلدینگ دانشمند معنا دارند
- موتور Playbook واقعی (چک‌لیست‌های گردش‌کار اجراشدنی، نه فقط نمایش لیست)
- انطباق: Compliance Export واقعی، Audit Log کامل، (در صورت نیاز مشتری دولتی) برچسب طبقه‌بندی و burn-on-read
- تست بار، تست نفوذ، سخت‌سازی امنیتی نهایی

### فاز ۴ — رشد و فروش (مستمر)
- Self-service onboarding برای سازمان جدید (بدون دخالت دستی تیم فنی)
- صورتحساب/اشتراک (Billing) برای مدل فروش به چند سازمان
- فعال‌سازی واقعی «تعامل بین‌سازمانی» (در صورت تقاضای بازار)
- اپ موبایل بومی (در صورت نیاز؛ تا این فاز PWA کافی است)

## ۲. نسخه‌ی فشرده برای محدودیت ۳ ماهه (در صورت نیاز واقعی به این بازه)

اگر ۳ ماه واقعاً غیرقابل‌تغییر است، scope را این‌طور کوچک کنید — نه کیفیت را پایین بیاورید:

1. فقط فاز ۱ قرارداد (شبکه اجتماعی کامل) + چندمستأجری پایه
2. برای ارتباطات: همان self-host کردن Mattermost (۳ تا ۵ هفته) به‌جای ساختن از صفر
3. برای مدیریت پروژه: به‌جای بازسازی کامل Mizito، یک ابزار متن‌باز آماده (مثل Plane یا Focalboard — که خود Mattermost هم دارد) را embed کنید
4. فاز ۲ و ۳ (دانش پیشرفته، قرارداد فناورانه، صندوق نوآوری) را رسماً به فاز بعدی قرارداد با کارفرما موکول کنید

---

## ۳. نقشه‌راه فرانت‌اند

1. مهاجرت از Vite SPA پروتوتایپ به **Next.js (App Router)** برای SSR در صفحات عمومی (لندینگ/SEO) و CSR در اپ
2. استخراج کامپوننت‌های `src/components/ui` به یک پکیج Design System داخلی (قابل نسخه‌گذاری، قابل استفاده در پروژه‌های دیگر شرکت)
3. اتصال واقعی به API با React Query (کش/retry/optimistic update) به‌جای `lib/api.ts` mock فعلی
4. real-time: WebSocket API رسمی Mattermost برای چت؛ یک کانال WebSocket مجزا (همان Ratchet موجود در Motoshub) برای اعلان/حضور بخش اجتماعی
5. تست: Vitest (unit) + Playwright (e2e روی مسیرهای حیاتی: ورود، ایجاد پست، ارسال پیام، ایجاد تسک)
6. PWA کامل (نصب‌پذیر، آفلاین-فرست برای فید و اعلان‌ها) به‌عنوان جای‌گزین موقت اپ بومی
7. بازبینی دسترس‌پذیری (a11y) و RTL edge-caseها قبل از انتشار نهایی

## ۴. نقشه‌راه بک‌اند

1. API Gateway واحد (Kong) جلوی سه ضلع: Oxwall/PHP (اجتماعی)، Node/NestJS (دانش/پروژه/تننت/گزارش)، Mattermost (ارتباطات)
2. Keycloak به‌عنوان IdP مرکزی: همه‌ی سه سرویس به یک‌جا اعتماد می‌کنند؛ SSO/LDAP هر سازمان فقط یک‌بار در Keycloak تعریف می‌شود، نه در هر سرویس جدا
3. نگاشت تننت موتوشاب ↔ Team در Mattermost (ایزولاسیون چت رایگان از طریق مفهوم بومی Mattermost)
4. چندمستأجری داده: shared-schema با `tenant_id` برای مشتریان کوچک، دیتابیس/کانتینر مجزا برای مشتریان حساس (بانک، نهاد دولتی)
5. صف پیام (RabbitMQ موجود) برای: ارسال پیامک/ایمیل، خروجی Compliance، ایندکس‌گذاری Elasticsearch، اجرای Playbook
6. مشاهده‌پذیری: Prometheus + Grafana + Loki برای متریک/لاگ همه‌ی سرویس‌ها؛ هشدار خرابی per-tenant
7. پشتیبان‌گیری و DR: snapshot دوره‌ای دیتابیس‌ها + تست بازیابی فصلی

## ۵. تیم‌بندی پیشنهادی (برای بازه‌ی ۶ تا ۷ ماهه)

| فاز | نقش‌های اضافه‌شده | جمع تیم |
|---|---|---|
| فاز ۰-۱ | ۱ Tech Lead، ۲ بک‌اند PHP/Oxwall، ۲ بک‌اند Node، ۱ DevOps/Platform، ۳ فرانت‌اند، ۱ طراح UI/UX، ۱ QA، ۱ PM | **۱۲ نفر** |
| فاز ۲-۳ | + ۱ بک‌اند (ماژول‌های تخصصی)، ۱ مهندس داده/جستجو (Elasticsearch + گزارش‌گیری)، ۱ متخصص امنیت/انطباق (پاره‌وقت) | **۱۵ نفر** |
| فاز ۴ (مستمر) | + ۱ مهندس Customer Success/Onboarding، ۱ SRE (پاره‌وقت) | **۱۷ نفر معادل تمام‌وقت در اوج** |

برای نسخه‌ی فشرده‌ی ۳ ماهه (بخش ۲ همین سند): همان تیم ۸ تا ۱۰ نفره‌ی پیشنهادی اولیه کافی است، چون بار اصلی (چت و مدیریت پروژه) به ابزارهای متن‌باز موجود سپرده می‌شود.

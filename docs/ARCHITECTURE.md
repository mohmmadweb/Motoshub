# موتوشاب — تحلیل نیازمندی، نقشه راه، تیم‌بندی و معماری

> منبع نیازمندی: قرارداد «استقرار و توسعه شبکه اجتماعی دانشمند (متوشاپ سازمانی)» بین هلدینگ دانشمند و شرکت دانش‌افراز فاخر ایرانیان (اردیبهشت ۱۴۰۵) + بررسی mizito.ir + بررسی کد منبع فعلی Motoshub (Oxwall/PHP).

---

## ۱. تحلیل صادقانه نیازمندی‌ها

### چیزهایی که در قرارداد **کم** است (باید قبل از فروش به چند سازمان اضافه شود)

1. **SSO/LDAP/Active Directory** — قرارداد فقط «نام‌کاربری/گذرواژه» و «ورود با موبایل» را پوشش می‌دهد. سازمان‌هایی مثل بانک ملت، وزارت علوم یا پلیس فتا تقریباً همیشه می‌خواهند کاربران را از AD/LDAP/SSO داخلی خودشان sync کنند، نه این‌که حساب جدید بسازند. این مهم‌ترین گپ برای فروش B2B است.
2. **مدل چندمستأجری (Multi-tenancy) و مدیریت اشتراک/صورت‌حساب** — در کل سند هیچ اشاره‌ای به این نیست که خود شما (فروشنده) چطور چند سازمان را روی یک پلتفرم مدیریت، مانیتور و billing می‌کنید. این دقیقاً هدف اصلی شماست پس باید جزو Phase 1 باشد، نه یک افزونه بعدی.
3. **White-labeling واقعی** (دامنه/برند/رنگ مستقل به ازای هر سازمان) — خواسته‌ی شما («کاربر نفهمد پشت صحنه چی هست») صریحاً در قرارداد نیامده.
4. **RBAC چندسطحی (Platform Super-Admin / Org-Admin / Group-Admin / Member)** — قرارداد فقط «مدیریت نقش‌ها و ناظم‌ها» در سطح یک سازمان را می‌گوید؛ برای SaaS باید یک لایه‌ی بالاتر (راهبر پلتفرم) اضافه شود.
5. **Audit Log و گزارش امنیتی قابل ارائه به نهادهایی مثل افتا** — قرارداد به «گواهی امنیتی افتا» اشاره می‌کند ولی audit trail مشخصی برایش تعریف نشده.
6. **لایه API رسمی / Webhook برای یکپارچه‌سازی با سیستم‌های سازمانی (ERP/CRM داخلی، اتوماسیون اداری)** — با وجود این‌که Composer پروژه فعلی از قبل `zircote/swagger-php` دارد (یعنی این قابلیت قبلاً پیش‌بینی شده ولی استفاده نشده).
7. **اعلان از طریق پیامک** — عجیب است که با وجود ده‌ها پلاگین SMS موجود در Motoshub (Kavenegar، Farapayamak، ICT و ...)، قرارداد فقط «اعلان برخط و رایانامه» را ذکر کرده. این یک ناهماهنگی بین زیرساخت موجود و scope مستندشده است.
8. **نسخه iOS بومی** — فقط Android ذکر شده. برای MVP اولویت کمی دارد؛ PWA می‌تواند جای‌گزین موقت باشد.
9. **نسخه‌گذاری اسناد (Document Versioning)** در ماژول مدیریت دانش — برای آرشیو قراردادها/مصوبات حیاتی است و در سند نیامده.
10. **امضای دیجیتال/الکترونیکی برای قراردادهای فناورانه** — با توجه به این‌که فاز ۳ مدیریت قرارداد فناورانه را پوشش می‌دهد، نبود امضای دیجیتال یک خلأ عملیاتی است.
11. **Onboarding خودکار سازمان جدید (self-service tenant provisioning)** — برای مدل «بفروشم به چند سازمان» باید بدون دخالت دستی یک سازمان جدید بتواند راه بیفتد.

### چیزهایی که در قرارداد **اضافه/خیلی‌خاص** است (برای MVP عمومی، نه برای فروش گسترده)

- **«صندوق‌های مالی نوآوری و شتاب‌دهی»** و **«مدیریت فرصت‌های پژوهشی»** (فاز ۳) کاملاً مختص دامنه‌ی کاری بنیاد مستضعفان/هلدینگ دانشمند هستند (سرمایه‌گذاری خطرپذیر، داوری طرح). اکثر مشتریان دیگر شما (بانک، دانشگاه، شرکت خصوصی) به این نیاز ندارند. **پیشنهاد: این دو را به‌صورت یک «ماژول اختیاری نصب‌شدنی» جدا بسازید، نه بخشی از core.**
- **مدیریت قراردادهای فناورانه با گردش داوری/مذاکره** هم تاحدی تخصصی است؛ می‌تواند به‌صورت یک ماژول عمومی‌تر «مدیریت قرارداد و تامین‌کننده» بازطراحی شود تا قابل فروش به سازمان‌های دیگر هم باشد.
- چت ویدیویی/کنفرانس آنلاین در Mizito دیده می‌شود ولی در قرارداد فقط «گفتگوی متنی» خواسته شده — اگر در MVP اضافه شود زمان را به‌خطر می‌اندازد؛ پیشنهاد می‌شود برون‌سپاری به یک سرویس آماده (مثلاً WebRTC SDK) به فاز بعد منتقل شود.

### تناقض زمانی مهم

ماده ۲ قرارداد مدت اجرا را **۶ ماه شمسی** تعریف کرده (با سه فاز)، اما خواسته‌ی شما این است که نسخه‌ی داخلی/فروشی خودتان در **۳ ماه** جمع شود. این دو با هم نمی‌خوانند اگر هدف، پوشش کامل سه فاز قرارداد باشد. رویکرد واقع‌بینانه:

> در ۳ ماه، **فاز ۱ کامل قرارداد + پایه‌ی چندمستأجری/برندسازی + لایه API یکپارچه** را تحویل بدهید (یعنی MVP قابل‌فروش به چند سازمان با هسته‌ی اجتماعی). فاز ۲ (پروفایل سازمانی + مدیریت دانش) و فاز ۳ (پروژه/قرارداد/صندوق) را به‌صورت ماژول‌های افزودنی در ماه‌های بعدی - با همان تیم یا تیم کوچک‌تر نگه‌داری - پیش ببرید. این هم با محدودیت ۳ ماهه شما سازگار است، هم خلاف قرارداد رسمی با کارفرما نیست (چون فاز ۲ و ۳ در قرارداد هم زمان جدا دارند).

## ۲. نقشه راه ۱۲ هفته‌ای (۳ ماه)

| هفته | محور اصلی | خروجی |
|---|---|---|
| ۱ | معماری پایه + Setup | API Gateway، تعریف schema چندمستأجری، CI/CD، اسکلت Next.js |
| ۲–۳ | لایه API روی هسته‌ی Oxwall | Endpoint های auth/users/groups/posts بر پایه‌ی `bol/` فعلی Motoshub + JWT bridge |
| ۴–۵ | فرانت‌اند هسته‌ی اجتماعی | فید، گروه، پروفایل، اعلان، گپ (متصل به API واقعی) |
| ۶ | Multi-tenant + White-label | Tenant resolver، تنظیم برند/دامنه، پنل راهبری پلتفرم |
| ۷ | SSO/Auth سازمانی + RBAC | SAML/OIDC connector، نقش‌های Platform/Org/Group |
| ۸ | انجمن + اعلان‌رسانی چندکاناله | Forum API+UI، اتصال به پلاگین‌های SMS موجود، صف RabbitMQ |
| ۹ | شروع فاز ۲ (نسخه‌ی نازک) | پروفایل سازمانی پایه + مدیریت دانش (آپلود/دسته‌بندی/جستجوی Elasticsearch) |
| ۱۰ | شروع فاز ۳ (نسخه‌ی نازک) | مدیریت پروژه پایه (لیست/بورد) — بدون Gantt کامل و بدون صندوق مالی |
| ۱۱ | QA، سخت‌سازی امنیتی، Audit log | تست نفوذ پایه، گزارش‌گیری، مهاجرت داده pilot سازمان اول |
| ۱۲ | Pilot Launch | استقرار on-premise/cloud برای هلدینگ دانشمند + آموزش کاربران (۲۴ نفر-ساعت طبق قرارداد) |

موارد زیر **آگاهانه از scope ۳ ماهه حذف و به فاز بعدی موکول می‌شوند**: Gantt کامل با dependency، صندوق نوآوری/شتاب‌دهی، مدیریت فرصت پژوهشی، امضای دیجیتال، اپ Android بومی، ویدیوکنفرانس.

## ۴. معماری

### اصل راهنما: «پنهان‌سازی درز بین Motoshub و Mizito پشت یک API Gateway»

```
                        ┌─────────────────────────┐
                        │   Next.js / React SPA    │   ← یک frontend واحد، مدرن
                        │ (هسته اجتماعی + پروژه)   │
                        └────────────┬─────────────┘
                                     │ HTTPS/JSON (یک دامنه: api.motoshub.app)
                        ┌────────────▼─────────────┐
                        │       API Gateway          │  (nginx / Kong)
                        │ مسیر یکپارچه‌سازی + auth   │
                        └──────┬───────────────┬────┘
                 /v1/social/*  │               │  /v1/work/*  ,  /v1/admin/*
        ┌────────────────────▼───┐   ┌─────────▼─────────────────┐
        │  PHP/Oxwall (Motoshub)  │   │   Node.js/NestJS سرویس‌ها   │
        │  + لایه‌ی API جدید روی    │   │  (پروژه، دانش، تننت، Billing)│
        │  bol/dao موجود          │   │                            │
        └───────────┬─────────────┘   └─────────────┬──────────────┘
                     │                                │
              MySQL/MariaDB (با tenant_id)    DB مستقل یا schema مجزا برای
              + Redis (cache/session)         سازمان‌های بزرگ/حساس (بانک ملت و …)
              + Elasticsearch (جستجو)
              + RabbitMQ (صف: SMS/ایمیل/اعلان)
              + Ratchet WebSocket (ریل‌تایم چت/حضور)
```

### بک‌اند

- **هسته‌ی فعلی (PHP/Oxwall) حفظ می‌شود** و فقط یک پلاگین جدید (`iisapi` یا مشابه) به آن اضافه می‌شود که Endpointهای REST/JSON روی `BOL`/`DAO` موجود برای users/groups/posts/forum/notifications/chat می‌سازد و سشن Oxwall را به JWT پل می‌زند (`OW_SessionAuthenticator` → `token_authenticator.php` موجود در `ow_core` دقیقاً برای همین کار طراحی شده).
- **سرویس‌های جدید (پروژه/دانش/تننت/Billing) در Node.js/NestJS یا حتی PHP Laravel جدا** نوشته می‌شوند تا توسعه‌ی سریع‌تر و تیم مستقل‌تر داشته باشند؛ هر دو پشت یک API Gateway قرار می‌گیرند تا frontend و مشتری نهایی هیچ‌وقت تفاوت را نبینند.
- **چندمستأجری**: مدل هیبریدی — سازمان‌های کوچک/متوسط: `tenant_id` در همان دیتابیس مشترک (هزینه پایین). سازمان‌های حساس/بزرگ (بانک ملت، پلیس فتا، نهادهای دولتی با نیاز data residency): دیتابیس/کانتینر مجزا (مطابق همان الگوی Docker Compose موجود در `docker-compose.yml`، فقط پارامتری‌شده به ازای هر تننت).
- **زیرساخت موجود که باید فعالانه استفاده شود** (در قرارداد نادیده گرفته شده ولی در composer.json از قبل هست): Elasticsearch برای جستجوی پیشرفته، RabbitMQ برای پردازش async پیامک/ایمیل/اعلان، Ratchet WebSocket برای حضور آنلاین و چت، `zircote/swagger-php` برای تولید خودکار OpenAPI spec از همان کنترلرهای PHP.
- **احراز هویت**: JWT کوتاه‌مدت + Refresh Token، با Adapter قابل‌تغییر برای SSO/SAML/OIDC به ازای سازمان‌هایی که AD دارند (Phase بعد از MVP).

### فرانت‌اند

- **Next.js (App Router) + TypeScript + Tailwind**، با همان زبان طراحی این پروتوتایپ (که با Vite ساخته شده چون برای GitHub Pages ساده‌تر بود؛ نسخه‌ی واقعی Next.js را برای SSR/SEO لندینگ و Edge caching می‌خواهد).
- **React Query** برای کش/همگام‌سازی داده با API، **Zustand** برای state سبک (notification badge، تننت فعال).
- **PWA** برای پوشش نیاز موبایل بدون توسعه‌ی اپ بومی در فاز اول.
- **سیستم برندسازی پویا**: رنگ/لوگو/فونت/دامنه از طریق یک Theme Provider که در زمان لاگین بر اساس تننت resolve می‌شود — دقیقاً منطقی که در صفحه‌ی `/app/admin` این پروتوتایپ شبیه‌سازی شده.

## ۵. قرارداد API (پایه‌ی واقعی روی همین مدل mock فعلی)

تمام مسیرها زیر `https://api.motoshub.app/v1` و با هدر `Authorization: Bearer <JWT>` + `X-Tenant-Id`.

```
POST   /auth/login                  ورود (username/phone + password یا OTP)
POST   /auth/refresh
POST   /auth/sso/callback           بازگشت از SSO سازمانی

GET    /users/me
GET    /users/:id
PATCH  /users/:id

GET    /groups
POST   /groups
GET    /groups/:id
GET    /groups/:id/members
POST   /groups/:id/join

GET    /posts?scope=feed|group&groupId=
POST   /posts
POST   /posts/:id/like
POST   /posts/:id/comments

GET    /forum/topics
POST   /forum/topics
GET    /forum/topics/:id/replies

GET    /chat/threads
GET    /chat/threads/:id/messages
WS     /chat/ws                     کانال WebSocket (Ratchet) برای پیام/حضور برخط

GET    /notifications
PATCH  /notifications/:id/read

GET    /knowledge/documents?category=
POST   /knowledge/documents
GET    /knowledge/search?q=

GET    /projects
POST   /projects
GET    /projects/:id/tasks
PATCH  /tasks/:id

# مسیرهای مخصوص راهبر پلتفرم (نقش platform-admin)
GET    /admin/tenants
POST   /admin/tenants
PATCH  /admin/tenants/:id/modules
PATCH  /admin/tenants/:id/branding
```

این فایل `src/lib/api.ts` در این مخزن همین قرارداد را با داده‌ی mock شبیه‌سازی می‌کند تا تیم فرانت‌اند بدون وابستگی به بک‌اند واقعی کار کند.
